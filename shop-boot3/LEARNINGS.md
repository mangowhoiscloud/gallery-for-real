# Learnings — Runtime Discoveries

### Learning: H2 2.x AUTO_INCREMENT syntax
- Context: Writing schema.sql for H2 in-memory test database (Spring Boot 3 ships H2 2.2.x)
- Discovery: `BIGINT AUTO_INCREMENT PRIMARY KEY` is rejected by H2 2.2.x. The parser expects `GENERATED BY DEFAULT AS IDENTITY` or similar standard SQL syntax.
- Rule: Always use `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` for auto-increment PKs in H2 2.x test schemas. This also works in PostgreSQL 10+, making it compatible with production.

### Learning: PasswordEncoder placement to avoid circular dependency
- Context: SecurityConfig (Item 6) needs PasswordEncoder, but AdminDataInitializer (Item 1) also needs it.
- Discovery: Defining PasswordEncoder in a separate AppConfig class (not SecurityConfig) allows Item 1 to seed admin data without SecurityConfig existing yet. SecurityConfig later injects the AppConfig-defined bean.
- Rule: Define PasswordEncoder in a standalone AppConfig, not inside SecurityConfig, when other components need it before SecurityConfig is created.

### Learning: ApplicationRunner timing relative to schema.sql
- Context: AdminDataInitializer seeds admin data using JdbcTemplate; schema.sql must run first.
- Discovery: Spring Boot's SQL initializer (spring.sql.init) runs during DataSource initialization (early in context creation), which completes before ApplicationRunner.run() is called. So ApplicationRunner can safely INSERT rows into schema-initialized tables.
- Rule: Use ApplicationRunner for startup data seeding; it always runs after spring.sql.init schema/data scripts.
